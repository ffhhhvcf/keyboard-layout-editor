## 🐛 报告问题

如果您发现了bug或有任何问题，请通过以下步骤提交issue：

1. **搜索现有issue**：在提交新issue之前，请先搜索现有issue，确保问题尚未被报告
2. **创建新issue**：
   - 选择合适的issue模板（如果有）
   - 标题：简明扼要地描述问题
   - 描述：
     - 详细说明问题的表现
     - 提供重现步骤
     - 说明预期行为和实际行为
     - 提供浏览器信息、操作系统版本
     - 如有可能，提供截图或视频
     - 提供相关代码示例

## ✨ 功能请求

如果您有新功能建议，请按照以下步骤提交：

1. **搜索现有功能请求**：确保您的建议尚未被提出
2. **创建新issue**：
   - 标题：使用"Feature Request: [功能名称]"格式
   - 描述：
     - 详细说明功能的用途和价值
     - 提供功能的设计思路
     - 如有可能，提供草图或示例
     - 说明该功能对用户的好处

## 🔧 开发流程

### 1. 克隆仓库

```bash
git clone https://github.com/ffhhhvcf/keyboard-layout-editor.git
cd keyboard-layout-editor
```

### 2. 创建分支

- 从`main`分支创建新分支
- 分支命名规则：
  - 功能分支：`feature/feature-name`
  - 修复分支：`fix/bug-description`
  - 文档分支：`docs/documentation-name`
  - 重构分支：`refactor/refactor-name`

```bash
git checkout -b feature/your-feature-name
```

### 3. 开发代码

- 遵循项目的[代码规范](#-代码规范)
- 编写清晰的代码注释
- 确保代码能正常运行

### 4. 测试

- 测试您的更改，确保功能正常
- 测试不同浏览器和屏幕尺寸
- 确保现有功能不受影响

### 5. 提交更改

- 提交信息应清晰、简洁
- 使用英文或中文，建议使用英文便于国际开发者理解
- 提交信息格式：
  ```
  类型(范围): 简短描述
  
  详细描述（可选）
  
  修复的issue #(issue编号)（可选）
  ```

- 类型包括：
  - `feat`: 新功能
  - `fix`: 修复bug
  - `docs`: 文档更新
  - `style`: 代码风格调整（不影响功能）
  - `refactor`: 代码重构
  - `perf`: 性能优化
  - `test`: 测试相关
  - `chore`: 构建过程或辅助工具的变动

```bash
git add .
git commit -m "feat: 添加新的键盘布局预设"
```

### 6. 推送分支

```bash
git push origin feature/your-feature-name
```

## 📝 代码规范

### HTML规范

- 使用语义化标签
- 缩进使用2个空格
- 属性值使用双引号
- 避免内联样式
- 确保标签正确嵌套

### CSS规范

- 使用BEM命名规范（可选，但建议）
- 缩进使用2个空格
- 选择器使用小写字母和连字符
- 避免使用!important
- 使用CSS变量管理颜色和尺寸
- 确保样式的响应式设计

### JavaScript规范

- 使用ES6+语法
- 缩进使用2个空格
- 变量和函数命名使用驼峰式
- 类名使用帕斯卡式
- 避免全局变量
- 使用const/let，避免var
- 箭头函数优先于普通函数
- 模板字符串优先于字符串拼接
- 异步代码使用async/await
- 编写清晰的注释，解释复杂逻辑

### 通用规范

- 保持代码简洁，避免冗余
- 代码可读性优先
- 避免过度优化
- 遵循DRY（Don't Repeat Yourself）原则
- 注释应解释"为什么"，而不是"是什么"

## 📥 提交Pull Request

1. 在GitHub上打开您的分支页面
2. 点击"New Pull Request"按钮
3. 填写PR信息：
   - 标题：简明扼要地描述PR的内容
   - 描述：
     - 详细说明PR的功能和改动
     - 说明解决的问题或实现的功能
     - 提供测试步骤
     - 如有可能，提供截图或视频
   - 关联相关issue
4. 点击"Create Pull Request"

## 🔍 审查流程

1. **自动检查**：PR将通过GitHub Actions进行自动检查
2. **人工审查**：项目维护者将审查您的PR
3. **反馈**：维护者可能会提出修改建议
4. **合并**：审查通过后，PR将被合并到主分支

### PR审查标准

- 代码质量：代码应符合项目规范，可读性好
- 功能完整性：实现了预期功能
- 无破坏性更改：不破坏现有功能
- 测试覆盖：功能经过充分测试
- 文档更新：如果需要，更新了相关文档
